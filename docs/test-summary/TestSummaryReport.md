### 1.4 测试风险

**计划识别风险**

- 数据库依赖：集成测试需要真实数据库环境
- 功能缺口：部分服务层缺少输入校验
- 时间风险：5 人分工，进度不一致可能导致集成困难

**实际发生风险**

| 风险 | 影响 | 处置 |
|-----|------|------|
| 服务层输入校验普遍缺失 | 大量测试失败（22 个），尤其是 OrderService（17 个） | 以 `assertThrows` 记录预期，`@Disabled` 标注已确认缺陷 |
| 状态机无前置约束 | confirmOrder/finishOrder/rejectOrder 状态机用例全部失败 | 记录为 BUG-005~007，集成测试同步标注 |
| 鉴权缺失 | 4 个高风险安全漏洞（BUG-017~020） | 集成测试明确记录，等待修复 |
| 测试进度延迟 | MessageService 和 OrderService 超计划完成 | 通过 PR 合并机制管理，总覆盖超出计划 |

**风险分析**

数据库依赖风险通过 `@WebMvcTest` + `@MockBean` 规避（所有单元测试和集成测试均不依赖真实数据库）。主要风险集中在被测代码本身的质量问题。

## 4 经验总结

### 4.1 亮点

**1. 编号与追踪体系完整**

用例编号规范（`UT-{模块}-{序号}`）+ `@DisplayName` 中文说明 + 缺陷 ID 引用，形成从用例设计文档 → 测试代码 → 缺陷日志的完整追踪链路。

**2. 用 @Disabled 规范标注已知缺陷**

失败测试不删除，以 `@Disabled("BUG-xxx: ...")` 明确标注缺陷编号，既保留测试意图，也防止 CI 误报，修复后直接删除注解即可启用。

**3. 明确区分功能缺口与代码 Bug**

部分被测方法根本未实现输入校验（如 `price < 0` 不拒绝），这属于功能缺口而非代码执行出错。对此类用例统一使用 `@Disabled` 标注并挂缺陷 ID，而非让测试持续红着，避免在 CI 中将已知功能缺口与真正的回归错误混淆。对于被测代码抛出的底层异常（如 DAO 层 NPE 直接透传），则视当前实现为预期行为，不额外断言，保持测试意图清晰。

**4. 黑盒与白盒结合**

MessageServiceImpl 实现了语句/条件/路径覆盖 100%；OrderServiceImpl 覆盖了正常路径、边界值、异常路径、状态机四类场景，质量高于单纯黑盒测试。

### 4.2 需要改进的点

**问题1：断言风格不统一**

- **现状**：混用 JUnit `assertEquals`（大多数测试）、AssertJ `assertThat`（部分）和 Hamcrest `assertThat`（MessageService）。
- **原因分析**：不同成员习惯不同，无统一规范约束。
- **改进建议**：统一采用 AssertJ (`assertThat`)，可读性更强，链式断言减少代码量。

**问题2：校验类用例集中在执行阶段才暴露缺口，测试介入时机偏晚**

- **现状**：22 个单元测试失败，根因集中在服务层缺少输入校验，而非测试用例设计有误。
- **原因分析**：测试用例在功能实现完成后统一编写，校验场景（如 `price < 0`、`null` 输入等）的缺口要到测试执行阶段才集中暴露，导致大批用例一次性失败，修复窗口已在计划末期。
- **改进建议**：在测试流程中引入"测试先行"：对关键校验场景先设计并编写失败用例（红），再验证对应实现是否到位，形成"用例设计 → 失败测试 → 实现验证 → 绿色通过"的闭环，使功能缺口在早期即可见，而非在集中执行阶段批量暴露。
