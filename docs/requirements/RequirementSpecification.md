# SoftwareTestingDemo 测试需求说明

## 1. 文档目的

本文档用于整理 SoftwareTestingDemo 场馆预约系统中与测试设计直接相关的业务需求，作为单元测试、集成测试、缺陷验证和测试总结的依据。

## 2. 测试范围

本次测试主要关注以下模块：

| 模块 | 测试关注内容 |
|---|---|
| 登录模块 | 登录成功、登录失败、Session 写入、未登录访问限制 |
| 场馆模块 | 场馆查询、新增场馆、场馆参数校验 |
| 订单模块 | 提交订单、查询订单、删除订单、订单审核、状态流转 |
| 留言模块 | 提交留言、查询留言、删除留言、留言审核、越权操作 |
| 新闻模块 | 新闻查询、新增新闻、新闻参数校验 |

## 3. 功能测试需求

### 3.1 登录模块

| 编号 | 测试需求 | 主要等价类/场景 |
|---|---|---|
| REQ-LG-01 | 用户使用正确账号和密码登录后，应写入 Session 并跳转到对应页面 | 正确账号 + 正确密码 |
| REQ-LG-02 | 用户使用错误密码登录时，应返回失败结果 | 正确账号 + 错误密码 |
| REQ-LG-03 | 未登录用户访问受保护功能时，应被拒绝或跳转登录 | 未登录访问订单/留言相关接口 |

### 3.2 场馆模块

| 编号 | 测试需求 | 主要等价类/场景 |
|---|---|---|
| REQ-VN-01 | 系统应支持分页查询场馆列表 | 合法页码、非法页码 |
| REQ-VN-02 | 系统应支持根据场馆编号查询场馆详情 | 存在的 venueID、不存在的 venueID、venueID 为 0 |
| REQ-VN-03 | 管理员新增场馆时，应保存场馆基本信息 | 合法场馆对象 |
| REQ-VN-04 | 新增场馆时应拒绝无效输入 | 场馆名为空、价格小于 0、场馆名重复 |

### 3.3 订单模块

| 编号 | 测试需求 | 主要等价类/场景 |
|---|---|---|
| REQ-OR-01 | 登录用户应能提交预约订单 | 已登录、场馆存在、hours > 0 |
| REQ-OR-02 | 提交订单时应拒绝无效参数 | hours <= 0、场馆不存在 |
| REQ-OR-03 | 登录用户应能查看自己的订单 | 已登录用户查询本人订单 |
| REQ-OR-04 | 删除订单时应校验登录状态和订单归属 | 未登录删除、删除他人订单 |
| REQ-OR-05 | 管理员应能审核待审核订单 | 待审核订单通过/驳回 |
| REQ-OR-06 | 订单状态流转应符合业务规则 | 已通过/已完成/已驳回订单再次审核 |

### 3.4 留言模块

| 编号 | 测试需求 | 主要等价类/场景 |
|---|---|---|
| REQ-MG-01 | 登录用户应能提交留言 | 已登录、content 非空 |
| REQ-MG-02 | 提交留言时应拒绝无效输入 | content 为空 |
| REQ-MG-03 | 用户应只能查询自己的留言 | 已登录查询本人留言、未登录查询 |
| REQ-MG-04 | 删除留言时应校验登录状态和归属 | 未登录删除、删除他人留言 |
| REQ-MG-05 | 系统不应允许用户伪造他人 userID 提交留言 | 请求参数 userID 与 Session 用户不一致 |
| REQ-MG-06 | 管理员应能审核待审核留言 | 待审核留言通过/驳回 |
| REQ-MG-07 | 留言状态流转应符合业务规则 | 已通过/已驳回留言再次审核 |

### 3.5 新闻模块

| 编号 | 测试需求 | 主要等价类/场景 |
|---|---|---|
| REQ-NW-01 | 系统应支持分页查询新闻列表 | 合法分页参数 |
| REQ-NW-02 | 系统应支持根据新闻编号查询新闻详情 | 存在的 newsID、newsID 为 0 |
| REQ-NW-03 | 管理员应能新增新闻 | title 非空、content 非空 |
| REQ-NW-04 | 新增新闻时应拒绝无效输入 | title 为空、content 为空 |

## 4. 重点业务规则

| 编号 | 规则 | 对应测试重点 |
|---|---|---|
| BR-01 | 后端不能只依赖前端校验，关键输入必须在服务端校验 | 空值、负数、重复名称 |
| BR-02 | 涉及写操作的接口应校验登录状态 | 未登录提交、删除、修改 |
| BR-03 | 涉及用户本人数据的接口应校验数据归属 | 删除他人订单/留言，伪造 userID |
| BR-04 | 订单和留言审核应遵守状态流转规则 | 非法重复审核、非法完成/驳回 |

## 5. 已知缺口与缺陷验证方向

现有测试和缺陷记录显示，以下内容需要作为重点缺陷验证方向：

| 编号 | 缺口说明 | 关联测试方向 |
|---|---|---|
| GAP-01 | 部分写接口缺少登录鉴权 | 未登录删除订单、未登录提交/删除留言 |
| GAP-02 | 部分接口缺少归属校验 | 伪造他人 userID、删除非本人数据 |
| GAP-03 | 部分服务层缺少输入校验 | 场馆名为空、价格为负、新闻标题/内容为空、留言内容为空 |
| GAP-04 | 状态机前置状态校验不足 | 订单和留言重复审核或非法流转 |

## 6. 测试方法建议

| 方法 | 适用内容 |
|---|---|
| 等价类划分 | 登录成功/失败、空值/非空值、合法/非法状态 |
| 边界值分析 | `id = 0`、`hours = 0`、`price < 0` |
| 错误推测 | 未登录直接调用接口、构造 HTTP 请求绕过前端校验 |
| 集成测试 | 登录后业务流程、页面跳转、JSON 响应、Controller 鉴权 |
| 单元测试 | Service 层参数校验、DAO 调用、状态流转逻辑 |
