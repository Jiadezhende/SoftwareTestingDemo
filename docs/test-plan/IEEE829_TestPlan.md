## 4. 待测特性

**业务功能**
- 用户认证：checkLogin、注册、注销
- 用户信息管理：查询、修改、删除
- 订单全生命周期：提交 → 审核通过/拒绝 → 完成
- 场馆信息：查询、分页、创建、修改、删除
- 新闻信息：CRUD
- 留言工作流：提交 → 审核通过/拒绝

**异常路径**
- 无效 ID 查询（不存在的记录）
- 状态非法时触发 RuntimeException（confirmOrder、rejectMessage 等）
- 未登录访问受保护路径（LoginException）

**集成端到端**
- 登录态传递（Session 写入与校验）
- 跨 Controller 业务流程的数据一致性

## 6. 测试方法与策略

**单元测试**
- 框架：JUnit 5 + Mockito（Mock DAO 层）
- 方法：等价类划分、边界值分析、决策表
- 覆盖目标：每个被测方法覆盖正常路径、异常路径、边界值各至少 1 条用例

**集成测试**
- 框架：Spring Boot Test + MockMvc
- 方法：场景法（完整业务流程）
- 数据隔离：每条测试前后清理数据库，使用 @Transactional 回滚

**自动化执行**
- 本地：`mvn test`
- CI：GitHub Actions（每次 push 自动触发，MySQL 8.0 Docker 容器）

## 8. 暂停与恢复准则

**暂停条件**
- CI 连续失败且原因未定位，阻塞其他成员提交

- 阻塞问题修复，CI 恢复绿色
- 测试环境重新可用并经验证

## 12. 测试风险

| 风险编号 | 风险描述 | 概率 | 影响 | 应对措施 |
|---------|---------|------|------|---------|
| R-01 | 本地与 CI 数据库配置不一致导致测试结果差异 | 中 | 高 | 统一使用 application.yml + CI 环境变量覆盖，每次本地运行前核对配置 |
| R-02 | 集成测试数据互相污染，导致用例依赖执行顺序 | 中 | 中 | 每条集成测试使用 @Transactional 自动回滚，或在 @BeforeEach/@AfterEach 清理数据 |
| R-03 | 成员进度不同步，关键路径模块延误 | 低 | 中 | 通过 check_progress.py 每日追踪，组长每周汇总进度 |
| R-04 | 复杂状态方法（confirmOrder、rejectMessage）边界用例覆盖不足 | 中 | 中 | 用例评审时检查状态机的全部转换路径 |
